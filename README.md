# x-clip-zh

把 X/Twitter 或网页链接，自动剪藏成中文 Obsidian Markdown 笔记的 OpenClaw skill。

它不是那种只会存个链接的半残工具，而是把正文抓下来、该翻译的翻译、该下图的下图、该落盘的落盘，最后还会检查是不是**真的写进了用户自己的 vault**。

## 它解决的不是“保存”，而是“保存得像样”

很多所谓剪藏，最后只是留下一个 URL，过几周再看已经废了。

这个 skill 解决的是另一件事：

- 把正文提取出来
- 非中文默认做忠实翻译
- 原文图片本地化保存
- 按阅读顺序嵌入 Markdown
- 最后校验文件和图片是否真实存在

也就是说，目标不是“记了一条链接”，而是得到一份以后还能搜索、阅读、引用、改写的 Obsidian 笔记。

## 适合谁

适合这几类人：

- 用 OpenClaw 管自己的知识库
- 平时把 X 长帖、英文文章、产品资料收进 Obsidian
- 不想只留链接，想把内容真正沉淀下来
- 希望 agent 能主动处理抓取、翻译、配图、落盘、校验这些脏活

## 主要能力

- 支持 X/Twitter 链接和普通网页链接
- 直抓失败时自动回退到 `r.jina.ai`
- 非中文内容默认做**严格、忠实翻译**
- 中文内容只做轻清洗，不乱改写
- 原文有图时，下载到本地 `Clippings/assets/`
- 使用 Obsidian 本地 embed，而不是远程图片热链
- 图片尽量按原文阅读节奏插入，而不是全堆顶部
- 写入后验证 Markdown、图片、embed 是否真实存在
- 所有笔记使用 YAML front matter 表头，记录来源、作者、日期、标签等信息

## 默认工作方式

这个 skill 的默认行为是：

1. 读取链接内容
2. 抓不到时走 `r.jina.ai` 兜底
3. 判断语言
4. 非中文默认做忠实翻译，不擅自改写成二创文章
5. 如有图片，下载本地并嵌入笔记
6. 自动尝试识别用户本机 Obsidian vault
7. 写入 `Clippings/` 和 `Clippings/assets/`
8. 最后做一次真实落地校验

## 使用前提

如果对方是自己的 OpenClaw 环境，通常满足下面这些条件时，拉下来就能直接用：

- agent 有 `read` / `write` / `web_fetch` / `exec` 这类基础能力
- 本机已经安装 Obsidian
- 本地至少存在一个可写的 vault
- agent 能读取本机的 Obsidian 配置
- agent 被允许往用户自己的 vault 写入 Markdown 和图片文件

说白了，它不是写死在作者机器上的私货逻辑，而是按“用户自己的 OpenClaw + 用户自己的 Obsidian”去跑。

## 什么时候不能硬跑

下面这些情况不要装懂，应该先问：

- 机器上有多个 vault，无法判断该写哪个
- 读不到 Obsidian 配置文件
- 用户不是 macOS，配置路径可能不同
- 用户根本不用 Obsidian
- 没有文件写权限
- 用户想保存到自定义目录，而不是默认 `Clippings/`

这时候正确做法不是瞎猜，而是让 agent 追问一个最小必要问题，比如：

- 你要写进哪个 vault
- 保存目录是不是 `Clippings/`
- 这次要忠实翻译，还是总结/改写

## 安装

目前这是一个 GitHub 分发版 skill。

最简单的方式是：

1. 把仓库拉到你自己的 skills 目录或可访问路径
2. 让 OpenClaw / 你的 agent 能读到 `SKILL.md`
3. 在使用时触发这个 skill

如果你后面要做成真正的一键安装版，可以再接 ClawHub 或你自己的安装脚本。

## 使用示例

用户可能会这样触发：

- 把这个推文存到 Obsidian
- 剪藏这条 X 长帖
- 把这个网页翻译后收进笔记
- 保存这篇文章到我的笔记库

期望输出通常包括：

- 一篇写进目标 vault 的 Markdown 笔记
- 一组下载到本地的图片文件（如果原文有图）
- Obsidian 本地 embed
- 一个完成报告：路径、来源、是否有图、摘要

## 表头格式

所有笔记统一使用 YAML front matter，格式如下：

```yaml
---
title: <中文标题>
source: <原始链接>
author: <作者，如已知>
published: <发表日期，YYYY-MM 或 YYYY-MM-DD>
clipped: <剪藏日期，YYYY-MM-DD>
tags:
  - <标签1>
  - <标签2>
created: <剪藏日期，YYYY-MM-DD>
---
```

不再使用内联 `**来源:**`、`**日期:**` 等写法，所有元数据统一放在表头。

## 输出约定

标准结果建议包含：

- 保存绝对路径
- 来源 URL
- 状态：`完成`
- 图片状态：`有图` / `无图`
- 一段约 200 字中文摘要

## 文件结构

- `SKILL.md`：skill 行为规范
- `README.md`：公开说明与使用方式

## 设计原则

这个 skill 有几个原则我故意卡得很死：

- 默认先忠实，再发挥，不要一上来就二创
- 图片是正文的一部分，不是装饰品
- 保存成功不算完成，**验证落盘成功**才算完成
- 能自动识别就别烦用户，真不确定再问
- 缺依赖就老实说，别假装已经做完
